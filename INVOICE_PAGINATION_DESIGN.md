# Invoice 페이지네이션 설계 문서

## 1. 페이지 분할 규칙

### 1.1 우선순위 규칙

페이지 분할은 다음 우선순위를 따른다:

1. **Header는 모든 페이지 상단에 반복된다**
2. **현재 페이지에 description(상품 리스트)을 가능한 한 채운다**
3. **페이지 하단에서 공간이 부족해질 때, 다음 페이지로 넘어가는 1순위는 footer다**
   - Footer는 분절 금지로 통째로 다음 페이지로 이동
4. **Footer가 렌더링에서 제외되면 DOM 흐름에 의해 summary가 자연스럽게 그 공간을 채운다**
   - 필요 시 spacing(마진/패딩) 규칙만 보장한다
5. **Footer를 제거해도 Summary가 페이지에 다 들어가지 않으면 Summary도 분절 금지로 통째로 다음 페이지로 이동한다**

### 1.2 결과 페이지 조합 (3가지 케이스)

#### 케이스 A: 1페이지에 다 들어가는 경우
```
1페이지: header + description + summary + footer
2페이지: 없음
```

#### 케이스 B: Description이 길어져서 Footer를 담을 수 없는 경우
```
1페이지: header + description + summary
2페이지: header + footer
```
- Footer가 다음 페이지로 통째로 이동한다
- **Footer-only 페이지에서는 footer는 페이지 하단 정렬을 유지한다(문서 느낌 유지)**

#### 케이스 C: Footer를 밀어낸 뒤에도 Summary까지 다 못 담는 경우
```
1페이지: header + description
2페이지: header + summary + footer
```
- Summary는 Footer 위에 위치한다

### 1.3 추가 결정 사항 (필수)

#### A. Footer-only 페이지의 "하단 정렬" 정의

**Footer-only 페이지에서는 footer를 고정하지 않고, 문서 흐름을 유지하되 시각적으로 하단에 가깝게 보이도록 최소 여백 규칙을 적용한다.**

- 페이지 컨테이너 하단에 고정 배치(absolute/flex bottom)하지 않음
- 일반 문서 흐름을 따르되, header와 footer 사이에 적절한 여백 유지
- **시각적 안정성을 위한 최소 상단 여백 규칙 적용** (운영 기준 11.3 참조)

#### B. 상품 리스트 행 분절 정책

**상품 리스트는 행(row) 단위 분절 금지: 다음 행이 들어가지 않으면 새 페이지로 넘긴다.**

- 한 행이 페이지 경계에 걸리면 해당 행 전체를 다음 페이지로 이동
- 테이블이 페이지 중간에서 끊기지 않도록 보장
- **이 정책은 절대 예외가 없다** (운영 기준 11.2 참조)

#### C. 측정 전용 상태 정의

**측정 전용 상태에서는 `.invoice-page`에 transform/transition을 비활성화한 CSS 클래스를 적용한 뒤 측정하고, 측정 완료 후 해제한다.**

- 측정 전: transform/transition 비활성화
- 측정 수행
- 측정 후: 스택 UI 효과(transform) 적용

#### D. 폰트 로딩 타이밍

**페이지 분할 측정은 폰트 로딩이 완료된 이후에 수행한다. 폰트 로딩 전에는 분할을 확정하지 않는다.**

- `document.fonts.ready` 또는 폰트 로딩 이벤트 대기
- 폰트 로딩 완료 전에는 임시 레이아웃만 표시하거나 로딩 상태 유지

## 2. 높이 계산 방식

### 2.1 기본 원칙

**페이지 분할 판정은 고정 상수 합산이 아니라 실제 렌더링된 DOM 높이를 기준으로 한다.**

#### 측정 전제

- **측정은 transform/애니메이션 적용 전 상태에서 수행한다**
- **스택 UI(transform)는 렌더링/측정이 끝난 후 적용한다**
- **측정 전용 상태에서는 `.invoice-page`에 transform/transition을 비활성화한 CSS 클래스를 적용한 뒤 측정하고, 측정 완료 후 해제한다**

#### 최우선 방법: 실제 렌더 트리에서 측정

- 실제 페이지 컨테이너 안에 렌더된 요소를 그대로 측정한다
- `getBoundingClientRect()` 또는 `offsetHeight` 사용
- 동일 컨테이너/동일 CSS/동일 폭/동일 미디어 상태에서 측정한다

#### 임시 요소 측정 (불가피한 경우에만)

- 임시 요소는 동일 조건(width, 클래스, CSS 적용, 미디어 상태)을 복제할 때만 신뢰 가능하다
- table/td 줄바꿈은 컨테이너 폭에 의존하므로 특히 주의한다

### 2.2 반응형별 높이 테이블 (Fallback/가드용)

반응형별 높이 테이블은 **주 로직이 아니라 fallback/가드로만** 사용한다.

- 실제 렌더링 측정이 불가능한 경우에만 사용한다

### 2.3 오차 허용 방식

**고정 tolerance(예: 20px)는 사용하지 않는다.**

대신 다음 규칙을 사용한다:
- **"다음 블록을 넣기 위한 최소 잔여 공간(minRemainingSpace)"** 규칙
- 또는 **안전 여백(safety margin)** 규칙
- (고정 px 값이 아니라 "블록별 최소 필요 공간" 개념으로 안정화한다)

## 3. Footer 이동 판정 기준

### 3.1 Footer 진입 비용(footerEntryCost)

Footer가 다음 페이지로 넘어가는 판정은 **footer 진입 비용**을 포함해 계산한다.

#### 핵심 정의

**footerEntryCost는 'footer를 넣기 위해 필요한 직전 간격 + footer 자체 간격 + footer 높이'의 합이다.**

해당 간격이 어느 요소에 정의돼 있든(앞 섹션/footer) 합산한다.

#### 판정 형태

- **남은 공간 ≥ footerEntryCost** 이면 footer를 현재 페이지에 포함 가능
- **남은 공간 < footerEntryCost** 이면 footer는 다음 페이지로 이동

### 3.2 구성 요소 예시 (1440px+ 기준)

#### 직전 간격 (footer 직전에서 필요한 간격) 예시
- description의 하단 spacing (예: margin-bottom + padding-bottom)

#### Footer 자체 간격 예시
- footer의 상단 spacing (예: padding-top + margin-top)

#### Footer 높이
- footer 내부 실제 DOM 높이 (실측)

---

## 4. Summary 위치 처리

### 4.1 Footer가 없을 때 Summary 위치

Footer가 렌더링에서 제외되면 DOM 흐름에 의해 summary가 자연스럽게 그 공간을 채운다.

- 별도의 "summary를 내리는" 특수 CSS에 의존하지 않는다
- 필요 시 spacing 규칙(요구되는 마진/패딩)만 보장한다

### 4.2 Summary 이동 판정

Footer가 빠진 상태에서 Summary가 페이지에 다 들어가지 않으면 Summary는 다음 페이지로 이동한다(분절 금지).

**Summary 이동 판정에 필요한 summaryCost는 다음을 포함한다:**
- Summary 위 간격 (앞 섹션의 bottom spacing 등, summary 진입에 필요한 직전 간격)
- Summary 자신의 margin/padding
- Summary 실제 높이

## 5. 구현 지시사항 (설계 수준)

### 5.1 "단일 itemRowHeight 가정" 금지 및 행 분절 정책

**itemRowHeight를 단일 상수로 가정하지 않는다.**

- 상품명 길이, 줄바꿈, 폰트 로딩, 반응형 폭에 따라 행 높이는 변동 가능하다
- 실제 분할은 현재 페이지에 아이템을 하나씩 추가(또는 동일 조건에서 가상 추가)하며 "페이지 높이 초과 여부"로 판정한다

**상품 리스트는 행(row) 단위 분절 금지:**
- 한 행이 페이지 경계에 걸리면 해당 행 전체를 다음 페이지로 이동
- 테이블이 페이지 중간에서 끊기지 않도록 보장

### 5.2 측정 대상이 없는 경우 (0 반환) 금지

특정 케이스에서 footer 또는 summary가 페이지에 존재하지 않으면, selector 기반 측정이 0을 반환할 수 있다.

**"없는 요소의 측정값 = 0"을 로직 근거로 삼지 않는다.**

- 측정은 각 섹션이 존재하는 페이지 조합에서만 수행하거나
- 존재하지 않는 섹션의 spacing/height는 별도 방식으로 확보한다 (정의된 기준값 또는 CSS 계산값 등)

### 5.3 순환 참조 방지 (측정 타이밍)

"페이지 분할 전에 측정값이 필요"하므로 측정 타이밍을 다음처럼 정의한다:

- **폰트 로딩이 완료된 이후에 측정을 수행한다** (`document.fonts.ready` 또는 폰트 로딩 이벤트 대기)
- **템플릿/기준 레이아웃(빈 페이지)을 먼저 렌더링한 뒤 측정한다**
- **아이템별 높이는 아이템을 추가하면서 판정한다**

### 5.4 Step 4 예외 처리 제거

Summary를 첫 페이지에 억지로 남기는 예외 처리는 우선순위 규칙과 충돌하므로 제거한다.

- 규칙: Footer가 빠지면 Summary가 흐름대로 재배치되고, 부족하면 Summary도 이동한다

## 6. 테스트 케이스

### 6.1 기본 케이스

#### 케이스 A
- **입력**: items 적음
- **출력**: 1페이지에 summary + footer 포함

#### 케이스 B
- **입력**: items 증가
- **출력**: footer가 2페이지로 이동, 2페이지는 footer-only (하단 정렬)

#### 케이스 C
- **입력**: items 매우 많음
- **출력**: 1페이지는 description만, 2페이지는 summary + footer

### 6.2 추가 필수 테스트 (실전 변수 반영)

#### 상품명 줄바꿈 테스트
- 동일 items 수에서 상품명 길이를 한 줄/두 줄로 바꿔 결과 케이스가 바뀌는지 검증

#### 반응형 브레이크포인트 테스트
- 같은 데이터로 1440px+, 1439px 이하, 768px 이하에서 결과 페이지 조합이 일관된 규칙으로 나오는지 검증
- (결과 케이스가 달라질 수는 있으나, 규칙 위반이 없어야 함)

#### Footer-only 페이지 정렬 테스트 (케이스 B의 2페이지)
- Footer가 페이지 하단 정렬을 유지하는지 검증

### 6.3 판정 기준 검증

**Footer 이동 판정 기준:**
```
남은 공간 < footerEntryCost
→ Footer는 다음 페이지로 이동
```

**Summary 이동 판정 기준:**
```
Footer가 빠진 상태에서:
남은 공간 < summaryCost
→ Summary도 다음 페이지로 이동
```

## 7. 화면 UX와 인쇄 분리

### 7.1 화면 UX (스택 애니메이션)

- 3D 종이 넘기기 효과
- 현재 페이지는 정면에 100% 노출
- 다음 페이지는 뒤에 쌓여 오른쪽 가장자리만 보임
- 화살표 버튼으로 페이지 전환

### 7.2 인쇄용 (@media print)

화면 UX와 별도로 인쇄용 CSS 필요:

- 측정/분할 로직은 "transform 없는 상태" 기준으로 수행
- 인쇄에서는 transform/opacity 등 화면 연출 제거, page-break 규칙 적용

```css
@media print {
    .invoice-page {
        page-break-after: always;
    }
    
    .invoice-page:last-child {
        page-break-after: auto;
    }
    
    /* 3D 효과 제거 */
    .invoice-page {
        transform: none !important;
        opacity: 1 !important;
    }
}
```

## 11. 운영 기준 (운영 중 흔들림 방지)

### 11.1 페이지 분할 재계산 트리거

**페이지 분할 재계산은 다음 경우에만 수행한다:**

1. **최초 렌더 + 폰트 로딩 완료 후 1회**
   - 초기 페이지 분할 확정

2. **viewport 리사이즈 시 재계산 (디바이스 전환급 변경만)**
   - 브레이크포인트 변경(768px, 1439px 경계) 감지 시
   - 스크롤, 사소한 레이아웃 변화에는 재계산하지 않음

**재계산을 제한하는 이유:**
- 스크롤/사소한 레이아웃 변화 때마다 페이지가 흔들리면 사용자 경험이 저하됨
- 성능 최적화: 불필요한 재계산 방지

### 11.2 행 분절 정책의 예외 금지

**행 단위 분절 금지 정책은 절대 예외가 없다:**

- **어떤 경우에도 row를 반으로 쪼개지 않는다**
- 공간이 부족하면 해당 행 전체를 다음 페이지로 넘긴다
- 테스트에서도 이 규칙을 강제 검증한다

**예외를 허용하지 않는 이유:**
- 인보이스 문서의 가독성과 전문성 유지
- 테이블이 페이지 중간에서 끊기면 문서 품질이 크게 저하됨

### 11.3 Footer-only 페이지의 최소 높이 규칙

**Footer-only 페이지의 시각적 안정성을 위한 최소 여백 규칙:**

- Footer-only 페이지에서 footer가 지나치게 "위에 몰려 보이지 않도록" 최소 상단 여백을 보장한다
- Header 하단 기준 최소 간격을 정의한다 (예: header 하단부터 footer 상단까지 최소 간격)
- 이는 디자인 룰이지만 결과물 품질에 직접적인 영향을 미친다

**구현 방법:**
- Footer-only 페이지에서 header와 footer 사이에 최소 여백을 CSS로 보장
- 또는 JavaScript에서 footer 위치를 계산할 때 최소 간격을 적용
